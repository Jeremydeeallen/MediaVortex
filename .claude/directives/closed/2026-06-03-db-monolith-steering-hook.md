## Outcome

The hook stops being a code-quality gate for `Repositories/DatabaseManager.py` and becomes an architecture-direction router. When a directive tries to add or modify a method in the monolith, the refusal reads forward ("place it in `Features/<X>/<X>Repository.py`") instead of backward ("fix these 12 preexisting violations first"). The desired end state is documented once, in a colocated feature doc the hook cites on every steer. Incremental migration becomes the default move on every DB-layer touch -- not a future big-bang migration. Operators and Claude both see the destination on every refusal; directive scope stays narrow; line count of `Repositories/DatabaseManager.py` becomes a strictly-monotone-decreasing invariant.

### Decisions Made

- **Reconciliation: criterion 1 vs `repository-split.feature.md` C4.** Directive criterion 1 enumerated "each repo inherits `Core/Database/BaseRepository`". The durable feature doc's criterion 4 forbids any `BaseRepository` superclass / repository inheritance ("Repositories compose `DatabaseService` via constructor injection"). The feature doc is the durable CONTRACT; the directive is the transient ASK. Resolution: the `## Perfect End State` section tracks the feature doc -- "per-aggregate repos colocated at `Features/<X>/<X>Repository.py`, composing `DatabaseService` (no inheritance)". The directive criterion is treated as satisfied when the synopsis enumerates the items it lists EXCEPT the `BaseRepository` line, since following both literally is impossible.
- **`## Perfect End State` shape.** Compact synopsis (one short paragraph + bulleted target shape) intended to be hook-cited from refusal text, not a duplicate of the feature doc's existing Success Criteria. Section anchor: `#perfect-end-state`; the R19 refusal text quotes this anchor verbatim.
- **Project-local command instead of upstream framework edit.** Initial plan was to extend the upstream `claude-rails` `/check-conformance` to be data-driven (read repo-local `.claude/.conformance-baselines.json`). After the first cross-repo edit landed but the second was REFUSED by the hook ("Editing a file outside the trusted repo ... is scope escalation beyond the MediaVortex directive"), the first edit was reverted and the implementation pivoted to a project-local command at `.claude/commands/mediavortex-check-baselines.md`. This honors the `feedback_hook_path_forward_is_the_answer` memory: when the hook says "this is scope escalation", do not invent a clever workaround. Net effect for the directive: criterion 5's verification path is `/mediavortex-check-baselines` instead of `/check-conformance`; mechanism and data-driven shape are equivalent.
- **Baseline counting semantics.** `line_count` uses PowerShell `(Get-Content).Count` semantics on Windows (5850), not `wc -l` (5849). `import_count` regex drops the `^` anchor so indented imports count -- this changes the count from 96 (anchored) to 123 (unanchored), a 27-line difference. The baseline file declares which semantic each metric uses; the project command spec matches.
- **R-rule slot R19.** R17 is the empty slot in the standards table but still appears in two archived directives; R19 is the next clean slot after R18. Function name `Test-R19-DatabaseManagerSteering` slots into the dispatcher idiomatically.
