## Outcome

A read-only audit script walks every legacy path-bearing column in the live PostgreSQL DB at `10.0.0.15:5432`, attempts `Path.FromLegacyString(legacy_value, storage_roots)` against each row, and produces a markdown report quantifying the parse-failure rate. Target: < 0.1% failures across all path-bearing columns. Every failure has a logged root cause (no matching prefix / validation reject / NULL / cross-platform shape). For rows where the typed pair is ALSO populated, the audit cross-checks that re-parsing the legacy value produces the same `(StorageRootId, RelativePath)` already stored -- catching backfill drift if any. After this directive, Phase 7 caller migration has a quantified confidence interval on the legacy-to-typed migration: how many rows actually round-trip cleanly versus how many will need a manual operator pass.

## Why now

Phase 5 verified the WIRE FORMAT works end-to-end on a 100-row sample per table. Phase 6 verifies the SCALE -- every row in production. Two distinct failure modes are surfaced here that Phase 5 cannot: (a) rows whose legacy `filepath` shape no `StorageRoots` prefix matches (operator forgot to add a prefix, or path was generated by an out-of-band script), and (b) rows where the backfill that populated the typed pair did NOT match what re-parsing the legacy column produces today (drift). Catching these now is dramatically cheaper than discovering them during Phase 7 when seven verticals are mid-migration.

## Acceptance Criteria

1. **Audit script exists.** `Scripts/PathMigrationRehearsal.py` exists. Read-only. Operator-runnable from the repo root: `py Scripts\PathMigrationRehearsal.py`.

2. **StorageRoots loaded once, longest-prefix-first.** Script SELECTs `(Id, CanonicalPrefix)` from `StorageRoots`, sorts by `length(CanonicalPrefix) DESC`, passes as the prefix list to every `Path.FromLegacyString` call. Documented in the script header.

3. **Per-table walk.** Script walks every path-bearing legacy column on the six tables: `MediaFiles.FilePath`, `MediaFilesArchive.FilePath`, `TranscodeQueue.FilePath`, `TranscodeAttempts.FilePath`, `TemporaryFilePaths.OriginalPath` + `LocalSourcePath` + `LocalOutputPath`, `ShowSettings.ShowFolder`. NULL values are counted but not parsed.

4. **Categorize each row.** For each non-NULL legacy value, attempt `Path.FromLegacyString(value, storage_roots)`. Categorize: `parsed`, `no_prefix_match`, `validation_rejected` (PathError from D9/D14/Phase 3 checks), `unexpected_error` (any other exception). Categories are mutually exclusive.

5. **Markdown report generated.** Output `path-migration-rehearsal-report-YYYY-MM-DD.md` at repo root with: per-(table, column) summary table (total rows / NULL / parsed / failed by category) + top 10 failure samples per category per (table, column).

6. **Parse-failure rate target < 0.1%.** Across all (table, column) audited, total failures / total non-NULL rows < 0.001. If exceeded, directive escalates -- root cause every cluster of failures before closing.

7. **Cross-check: re-parsed legacy matches stored typed pair.** For rows where BOTH the typed pair and the legacy column are populated, the audit calls `FromLegacyString(legacy)` and compares the result against `Path.FromRow(row)`. Any mismatch is logged as a `cross_check_drift` finding. Target: zero drift. Drift indicates a backfill defect.

8. **Workers may keep running.** Script issues only `SELECT` statements; no transactions held open beyond a single query. Documented as a one-line script-header note. (Phase 5 needed worker bracket because of INSERT; Phase 6 does not.)

9. **Runtime ceiling.** Script completes in < 5 minutes wall time on the operator's I9 against the current data volume (~100K rows total across the six tables). If it exceeds the ceiling, profile and report; do not abort the directive.

10. **No code change to `Core/Path/Path.py`.** Audit exercises the existing API.

## Engineering Calls Already Made

- **Cross-check (C7) is the most valuable diagnostic.** Phase 5 proved the wire format works; Phase 6's cross-check answers "did the backfill that populated the typed pair actually do the right thing?" If the cross-check finds drift, the typed-pair column values cannot be fully trusted as canonical until those rows are re-parsed. This is the single test that should be most carefully implemented; the per-row failure audit is mechanical.

- **No pagination needed at current scale.** MediaFiles has ~50K rows; total across all six tables ~100K. At 200 bytes per legacy string in memory, total ~20MB. Single `SELECT` per table fits in RAM. The script docstring notes "if growth exceeds 1M rows on a single table, switch to server-side cursor or LIMIT/OFFSET pagination" -- forward-looking note, not current need.

- **Single script vs per-table scripts.** One script that walks all six tables. Operator runs it once, gets one report. Per-table scripts would clutter the Scripts/ directory and require operators to run six commands.

- **Where the report lives.** Repo root (`path-migration-rehearsal-report-YYYY-MM-DD.md`). Visible after `git status`. Committed with the directive close. Future Phase 8 can grep this report to confirm migration completeness before dropping legacy columns. Alternative: `docs/` or `Reports/` -- but the repo has no existing convention for ad-hoc reports, and repo root is the lowest-friction default.

- **Cross-check matches what semantics?** `FromLegacyString(legacy)` returns a `Path` with `(NewSid, NewRel)`. The row's stored typed pair is `(StoredSid, StoredRel)`. Match iff `NewSid == StoredSid AND NewRel == StoredRel` byte-exact. Case-insensitivity in `FromLegacyString` prefix matching may produce a different `RelativePath` casing than the stored one -- so the comparison must respect that the stored value was canonicalized at backfill time. The audit notes both the legacy value and the stored typed pair on every drift finding so root cause is diagnosable.

- **What counts as "validation_rejected" vs "unexpected_error".** `validation_rejected` is `PathError` from any of the D9/D14/Phase 3 construction checks (NUL, control char, Win32 namespace, UNC, leading separator, drive-letter, `..`). Or from `FromLegacyString`'s own checks (empty input, no-match-prefix). `unexpected_error` is any other exception (e.g., a psycopg2 type-coercion bug). Distinguishing these matters because validation rejects are likely false positives (legacy values that are technically illegal but happened to be ingested by the scanner) while unexpected errors are infrastructure bugs.

- **Worker concurrency safety.** Read-only `SELECT` queries against tables workers also read. PostgreSQL's MVCC isolation means the audit reads a consistent snapshot; concurrent worker writes don't affect the audit and vice versa. No bracketing needed.

## Escalation Defaults

- If parse-failure rate > 0.1% -> investigate root cause clusters before closing. If failures are concentrated in a fixable class (e.g., one missing StorageRoots row would fix 5%), document the fix-action in Findings + Promotions; do not close until the path forward is named.
- If cross-check drift > 0 -> escalate. Drift means the typed pair is not equal to the re-parsed legacy form. Could indicate (a) the legacy column was edited after backfill, (b) the backfill bug populated the wrong canonical, or (c) FromLegacyString semantics changed between backfill time and now. Each of these has a different remediation.
- If the audit script crashes mid-run -> investigate; likely psycopg2/encoding edge case on a specific row. Add a try/except per row to keep going, log the exception, count as `unexpected_error`.
- If runtime exceeds 5 min -> profile; do not abort. Add a NOTE to Findings.
- Risk tolerance: medium. False-negatives (audit passes but a real migration defect exists) are the main risk; false-positives (audit flags a row that's actually fine) are the recoverable failure mode.

- **Six content-drift rows.** Real anomaly: legacy `FilePath` and typed pair `(StorageRootId, RelativePath)` point at materially different files. Examples: MediaFiles id=22899 legacy says `.mp4` while typed-pair says `.avi`; MediaFilesArchive ids 54894/54969/58925 etc. show resolution changes (`2160p` vs `720p`) between legacy and typed pair. Pattern: post-replacement or post-transcode updates that touched ONE of the two columns but not the other. Root-cause hypothesis: `Features/FileReplacement` or `Features/TranscodeJob` updates `MediaFiles.FilePath` but not `MediaFiles.StorageRootId`/`RelativePath` -- the typed pair was backfilled at a snapshot in time and never re-synced when files were replaced. This is a real bug class but only 6 affected rows out of 101,355.
- **49 case-only drift rows (informational).** All in MediaFilesArchive `Z:\videos\Couple\...` legacy vs stored `Videos/Couple/...`.
